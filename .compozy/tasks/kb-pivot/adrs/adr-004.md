# ADR-004: Firecrawl REST API for URL Scraping

## Status

Accepted

## Date

2026-04-11

## Context

The `ingest url` command needs to scrape web pages and convert them to Markdown. The karpathy-kb skill currently delegates to firecrawl (a cloud/self-hosted web scraping service). Firecrawl does not have a Go SDK — only Python and Node.js. We need to choose between calling the REST API directly or shelling out to the firecrawl CLI.

## Decision

Integrate with firecrawl via their REST API (`POST /v2/scrape`) using Go's `net/http`. The API key is stored in config (`FIRECRAWL_API_KEY` env var or `[firecrawl] api_key` in TOML). The API URL defaults to `https://api.firecrawl.dev` but is overridable for self-hosted instances.

## Alternatives Considered

### Alternative 1: Shell-out to firecrawl CLI

- **Description**: Call `firecrawl scrape <url>` as a subprocess; users install via `npm i -g firecrawl`
- **Pros**: Simple Go code, CLI handles auth/config
- **Cons**: Requires npm/Node.js runtime, harder error handling, output parsing
- **Why rejected**: Adds Node.js dependency; REST API is cleaner and more controllable

### Alternative 2: Native Go scraping (colly + html-to-markdown)

- **Description**: Fetch HTML with net/http, parse with goquery, convert with html-to-markdown
- **Pros**: No external dependency, works offline
- **Cons**: Doesn't handle JS-rendered sites, no anti-bot bypass, lower quality than firecrawl
- **Why rejected**: Firecrawl handles JS rendering, anti-bot measures, and produces cleaner output; native scraping can be a fallback for simple pages

## Consequences

### Positive

- Clean HTTP integration with Go's standard library
- Works with both cloud and self-hosted firecrawl instances
- Proper error handling and response parsing
- No npm/Node.js dependency

### Negative

- Requires firecrawl API key (paid service or self-hosted)
- Network dependency for URL ingestion
- Need to handle rate limiting and API errors

### Risks

- API changes — mitigate by pinning to v2 endpoint
- Service unavailability — could add native Go scraping as fallback in future

## Implementation Notes

```go
type FirecrawlClient struct {
    apiKey  string
    apiURL  string
    httpClient *http.Client
}

func (c *FirecrawlClient) Scrape(ctx context.Context, url string) (*ScrapeResult, error)
```

Request: `POST /v2/scrape` with `{"url": "...", "formats": ["markdown"]}` and `Authorization: Bearer <key>`.

Config:
- `FIRECRAWL_API_KEY` env var
- `FIRECRAWL_API_URL` env var (default: `https://api.firecrawl.dev`)
- TOML: `[firecrawl]` section with `api_key` and `api_url`
