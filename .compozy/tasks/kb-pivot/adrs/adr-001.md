# ADR-001: Topic-Centric CLI Command Taxonomy

## Status

Accepted

## Date

2026-04-11

## Context

The kodebase CLI is pivoting from a codebase-only vault generator to a general-purpose knowledge base tool (Karpathy KB pattern). The current command surface (`generate`, `inspect`, `search`, `index`, `version`) is codebase-centric and does not accommodate multiple ingest source types, topic management, or lint operations. We need a new taxonomy that groups commands logically around the knowledge base lifecycle while keeping codebase analysis as one ingest source among many.

## Decision

Adopt a topic-centric command taxonomy where `topic` is the primary organizing concept and `ingest` uses subcommands per source type:

- `kb topic {new,list,info}` — scaffold and manage topics
- `kb ingest {url,file,youtube,codebase,bookmarks}` — ingest from various sources
- `kb lint [<slug>]` — structural health checks
- `kb index [<slug>]` — QMD collection management
- `kb search <query>` — semantic/lexical search
- `kb inspect {smells,dead-code,complexity,...}` — codebase-specific analysis (works on codebase-ingested topics)
- `kb version` — build info

The binary is renamed from `kodebase` to `kb` (see ADR-002).

## Alternatives Considered

### Alternative 1: Flat verbs

- **Description**: All commands at top level (`kb new`, `kb ingest`, `kb lint`). Source type as a flag `--type url|file|youtube`.
- **Pros**: Simpler surface, fewer nested commands
- **Cons**: Less discoverable; `--type` flag is awkward for fundamentally different operations (URL scraping vs file conversion vs YouTube transcript)
- **Why rejected**: Source types have different required arguments and behavior — subcommands map better than flags

### Alternative 2: Phase-aligned

- **Description**: Mirror karpathy-kb's 4-phase pipeline directly: `ingest`, `lint`, `index`, `search` plus `topic`.
- **Pros**: Most aligned with the knowledge base mental model
- **Cons**: Auto-detection of source type adds complexity and error surface; no room for codebase-specific inspect subcommands
- **Why rejected**: Loses discoverability of ingest source types and doesn't accommodate the rich codebase analysis pipeline

## Consequences

### Positive

- Clear command grouping that scales with new ingest sources
- Each ingest subcommand can have source-specific flags and validation
- `inspect` continues to serve codebase-specific queries without polluting general KB commands
- `kb ingest --help` shows all available source types at a glance

### Negative

- Deeper command nesting (`kb ingest youtube <url>`) vs flat (`kb ingest <url> --type youtube`)
- Adding a new ingest source requires a new subcommand file

### Risks

- Command proliferation if too many ingest sources are added — mitigate by keeping each subcommand thin (adapter pattern)

## Implementation Notes

- Cobra's nested command structure handles this naturally with `ingest` as a parent command
- Each ingest subcommand delegates to a shared pipeline: fetch/convert → add frontmatter → write to `raw/` → update log
- `inspect` subcommands only work on topics that have codebase-ingested data
