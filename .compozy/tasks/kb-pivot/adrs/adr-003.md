# ADR-003: Native Go Document Conversion with Converter Registry

## Status

Accepted

## Date

2026-04-11

## Context

The knowledge base CLI needs to convert many document formats (PDF, HTML, DOCX, PPTX, XLSX, EPUB, CSV, JSON, XML, images) to Markdown for ingestion into the vault. The Python tool `markitdown` by Microsoft handles this well but adding a Python runtime dependency conflicts with the single-binary Go design. We need a format conversion strategy that maximizes coverage while keeping the tool self-contained.

## Decision

Implement format conversion natively in Go using a **converter registry pattern** (inspired by markitdown's architecture). Each format has a dedicated converter implementing a common interface. Key libraries:

| Format | Library | License |
|--------|---------|---------|
| PDF | pdfcpu/pdfcpu | Apache-2.0 |
| HTML | JohannesKaufmann/html-to-markdown v2 | MIT |
| XLSX | xuri/excelize | BSD-3 |
| DOCX/PPTX | archive/zip + encoding/xml (direct XML parsing) | stdlib |
| EPUB | archive/zip + encoding/xml + html-to-markdown | stdlib + MIT |
| CSV/JSON/XML | encoding/csv, encoding/json, encoding/xml | stdlib |
| Images (OCR) | otiai10/gosseract (Tesseract bindings) | MIT |
| YouTube | kkdai/youtube (transcript extraction) | MIT |

For DOCX and PPTX, follow markitdown's approach: treat them as ZIP archives containing XML and parse the XML directly rather than depending on heavy Office libraries.

## Alternatives Considered

### Alternative 1: Shell-out to markitdown

- **Description**: Call `markitdown` as a subprocess, requiring users to `pip install markitdown[all]`
- **Pros**: Maximum format coverage, proven quality, minimal Go code
- **Cons**: Python runtime dependency, breaks single-binary promise, version management complexity
- **Why rejected**: Conflicts with the Go single-binary design philosophy

### Alternative 2: Hybrid (native for simple, markitdown for complex)

- **Description**: Native Go for HTML/CSV/JSON/XML, shell-out to markitdown for PDF/DOCX/PPTX
- **Pros**: Best quality per format
- **Cons**: Still requires Python for the most common formats, complex routing logic
- **Why rejected**: Half-measures — still has the Python dependency for the formats users need most

## Consequences

### Positive

- Single binary, no external runtime dependencies (except Tesseract for OCR, which is optional)
- Each converter is independently testable
- New format support is added by implementing the Converter interface
- ZIP+XML approach for Office formats is proven (markitdown uses the same pattern)

### Negative

- PDF text extraction quality from pdfcpu may be lower than pdfminer (Python)
- DOCX/PPTX parsing requires implementing XML walkers for each format
- OCR requires Tesseract system dependency (optional feature)

### Risks

- Complex PDFs (scanned, multi-column) may extract poorly — mitigate by supporting a `--ocr` flag that uses Tesseract on PDF pages
- Office format coverage may miss edge cases — mitigate by testing against real-world documents

## Implementation Notes

Core converter interface:

```go
type Converter interface {
    Accepts(ext string, mimeType string) bool
    Convert(ctx context.Context, r io.ReadSeeker, info FileInfo) (*ConvertResult, error)
}

type ConvertResult struct {
    Markdown string
    Title    string
    Metadata map[string]any
}
```

Registry walks converters in priority order, first match wins.
